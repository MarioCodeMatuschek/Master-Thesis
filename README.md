
# Virtual 2D LiDAR Dataset Generator (Python)

This application generates synthetic 2D LiDAR scans for complex indoorâ€‘like environments. It supports two **scenario layouts**:
- **Union**: overlapping axis-aligned rectangles (rooms/walls) forming a connected free space.
- **Apartment**: BSP (binary space partitioning) floorplans â€” the map is recursively split along the longer axis into rooms, with doors between adjacent rooms.

It also supports:
- **Fully automated** dataset creation (random scenarios, poses, and paths)
- **Manual** start/goal specification (CLI and GUI)
- **Single-scan CSV** (long format, one row per ray)
- **Sequence JSON** (for navigation/temporal models)
- **Scenario visualization** (layout-aware: segment lines for union, rooms + doors for apartment)

All automatically sampled and manually chosen poses are constrained to lie
inside the free space (union of rectangles or apartment interior) and not on any wall/segment.

## Quick start

**Choosing layout:** Use `--layout union` (default) for overlapping rectangles, or `--layout apartment` for BSP floorplans with rooms and doors. For apartment layout you can set `--apt_iterations N` (default 3) to control how many rooms are created.

```bash
# Example: automated generation of 10 scenarios (union layout, default)
python3.12 -m v2dlidar.cli auto \
  --out ./dataset \
  --layout union \
  --scenarios 10 \
  --scans_per_scenario 500 \
  --headings_per_pose 4 \
  --seq_per_scenario 10 \
  --seq_steps 50

# Example: automated generation with apartment layout (BSP floorplan)
python3.12 -m v2dlidar.cli auto \
  --out ./dataset \
  --layout apartment \
  --apt_iterations 3 \
  --scenarios 10 \
  --scans_per_scenario 0 \
  --seq_per_scenario 0 \
  --seq_steps 0

# Example: manual one scenario with start/goal (auto-generated paths)
python3.12 -m v2dlidar.cli manual \
  --out ./dataset \
  --scenario_id 0 \
  --start 2 2 \
  --goal 25 25 \
  --scans_per_scenario 500 \
  --seq_steps 50


## Outputs and formats

For each scenario folder `OUT_DIR/scenario_XXXX/` the following files are produced:

| File or folder | Description |
|----------------|-------------|
| `scenario_meta.json` | Scenario and LiDAR parameters (spec + lidar). |
| `scans_long.csv` | Long-format table: one row per LiDAR ray (all scans concatenated). |
| `sequences/seq_XXXX.json` | One file per navigation sequence (A* path + LiDAR at each step). |
| `scan_paths/scan_XXXX.json` | One file per scan: ideal A* path from start to goal for that scan. |

### scenario_meta.json

Top-level keys:

**spec** (ScenarioSpec):

- `width`, `height`: map size in meters.
- `n_rects`: number of rectangles used to build the union (for `layout: "union"`).
- `rect_min`, `rect_max`: 2-tuples `[width, height]` in meters for rectangle size ranges.
- `wall_thickness`: used when rasterising walls.
- `seed`: RNG seed that uniquely defines this scenario layout.
- `outline_res`: grid resolution (meters per cell) used for building the outline.
- `layout`: `"union"` (random overlapping rectangles) or `"apartment"` (BSP floorplan with rooms and doors).
- **Apartment layout** (when `layout` is `"apartment"`): the map is generated by recursive BSP splitting. Each split creates two rooms and a door between them.
  - `apt_iterations`: number of BSP split iterations (default 4); more iterations yield more rooms.
  - `apt_min_door_width`: minimum door gap in meters (default 0.6); doors are traversable after wall inflation.
  -  BSP generation uses `apt_iterations` only.

**lidar** (LidarSpec):

- `fov_deg`: field of view in degrees (typically 360).
- `num_rays`: number of beams per scan (typically 720).
- `max_range`: maximum measurable range in meters.
- `range_noise_std`: standard deviation of Gaussian range noise.
- `dropout_prob`: probability that a valid beam is reported as a miss.

### scans_long.csv

Long-format table where each row = one LiDAR ray (one scan has `num_rays` rows).

**Columns:**

- `scenario_id`: integer ID of the scenario (matches folder index).
- `scan_id`: integer ID of the scan within this scenario.
- `t`: time stamp (currently always 0.0, reserved for temporal use).
- `pose_x`, `pose_y`, `pose_yaw`: robot position and heading in meters/radians when this scan was taken.
- `goal_x`, `goal_y`: goal position in meters for the path associated with this scan.
- `theta_deg`: angle of this individual ray in degrees (0â€“360).
- `r_m`: measured range along the ray (with noise and dropout), in meters.
- `hit_x`, `hit_y`: world coordinates of the hit point in meters; empty if no valid hit.
- `valid`: 1 if there is a finite hit, 0 if not.
- `noise_free_r_m`: ideal range without noise/dropout, in meters.

For a given `(scenario_id, scan_id)`, the fields `pose_x`, `pose_y`, `pose_yaw`, `goal_x`, and `goal_y` are constant across all `num_rays` rows.
All scans that share the same `(scenario_id, pose_x, pose_y, goal_x, goal_y)` come from the **same navigation task** (same start position and goal) but with different initial headings; only `pose_yaw` and the LiDAR measurements differ between those scans.

### sequences/seq_XXXX.json

Each file describes one navigation sequence (A* path from start to goal) with LiDAR scans at each step.

**Top-level fields:**

- `scenario_id`: integer ID of the scenario.
- `start`: `{ "x": float, "y": float }` start position in meters.
- `goal`: `{ "x": float, "y": float }` goal position in meters.
- `manual`: optional; if present, `true` for manually created sequences (GUI / manual CLI). Automated sequences do not include this key.
- `steps`: list of per-time-step records, each with:
  - `step`: integer index (0 to seq_steps-1).
  - `pose_x`, `pose_y`, `pose_yaw`: robot pose at this step (meters, radians).
  - `theta_deg`: list of per-ray angles for the scan at this step.
  - `ranges`: list of per-ray ranges (with noise and dropout) in meters.

### scan_paths/scan_XXXX.json

One file per scan that had a valid A* path. `scans_per_scenario` controls how many **poses** are sampled in a scenario, and `headings_per_pose` controls how many **scans (headings)** are taken at each pose, so the number of `scan_XXXX.json` files per scenario is:

- `num_scan_files = scans_per_scenario Ă— headings_per_pose`

Each file links one `(scenario_id, scan_id)` in `scans_long.csv` to its planned path.

**Fields:**

- `scenario_id`: integer scenario ID.
- `scan_id`: integer scan ID (matches rows in `scans_long.csv`).
- `start`: `{ "x", "y", "yaw" }` start pose in meters and radians.
- `goal`: `{ "x", "y" }` goal position in meters.
- `path`: list of `{ "x", "y" }` points in meters along the ideal A* path from start to goal.

## Sequences and `seq_steps`

Two main data types:

- **Single scans** (`scans_long.csv`):
  - Each `scan_id` is one LiDAR sweep at a fixed pose.
  - Good for static tasks (range prediction, occupancy mapping, obstacle detection).

- **Sequences** (`sequences/seq_XXXX.json`):
  - Each file is a trajectory from a start to a goal inside a scenario.
  - The path is computed with A*, then subsampled to `seq_steps` points.
  - At each point, a LiDAR scan is simulated and stored as a `step`.

`seq_steps` controls **how many samples you take along a fixed path**:

- Larger `seq_steps` â†’ more steps per sequence, finer sampling, longer sequences.
- Smaller `seq_steps` â†’ fewer steps, coarser sampling, shorter sequences.

## Manual GUI (pick start/goal and append to dataset)
You can launch a Tkinter + Matplotlib GUI to visualize the scenario, pick a **Start** (left click) and **Goal** (right-click or Control+click on Mac), and append the resulting **manual sequence** and **single-capture CSV** to the dataset folder.

```bash
python3.12 -m v2dlidar.gui_manual
```

- **Layout**: choose **Union** (overlapping rectangles) or **Apartment** (BSP floorplan). Generation logic and visualization are separate for each layout.
- **Union**: scene is drawn as wall segments. Parameters include width/height, # Rectangles, seed.
- **Apartment**: scene is drawn as **rooms** (filled rectangles) and **doors** (gaps + swing arcs). Parameters include **Apartment iterations** (BSP splits, default 4), Apartment rows/cols/door prob (kept for compatibility).
- **Output dir** and **Scenario ID**: where to write `scenario_XXXX/` and which ID to use.
- **Seq steps**: number of steps along the A* path when appending a manual sequence (and for path preview).
- Use **Generate/Refresh** to regenerate the scene (respects layout and seed).
- **Complexity** panel: segment count and wall density.
- After picking Start and Goal, click **Append to Dataset (Manual)**. Files are saved under the chosen output dir:
  - `sequences/seq_XXXX_manual.json` (next free index)
  - `single_capture.csv` (720 rows @ 0.5Â°)

Start/Goal and A* path work the same for both layouts; occupancy and pathfinding use the same segment-based pipeline.

### Path preview toggle
Enable/disable **Preview planned path** to overlay the ideal A* path between Start and Goal before saving. The preview recomputes when you pick Start/Goal or regenerate the scenario. **Seq steps** controls how many points are shown along the path (and how many are written when appending). All clicks for Start/Goal are constrained to the free interior (off walls and inside the map).


### Scenario Visualization

After generating a dataset you can visualize any scenario. Drawing is **layout-aware**:
- **Union** scenarios: walls are drawn as segment lines.
- **Apartment** scenarios: walls are drawn as **rooms** (filled rectangles with labels) and **doors** (gaps and swing arcs), matching the Manual GUI.

```bash
# Walls (and layout) + first scan pose
python3.12 -m v2dlidar.visualize_scenario --root ./dataset_10x1 --scenario_id 0

# Walls + ideal path + start/goal for a specific scan (from scan_paths/scan_XXXX.json)
python3.12 -m v2dlidar.visualize_scenario --root ./dataset_10x1 --scenario_id 0 --scan_id 0

# Layout + LiDAR parameterization (single noisy scan)
# Left: walls, rooms/doors, pose, path, and rays in world coordinates
# Right: robot-centric polar view (0Â° = forward) of noisy vs. noise-free ranges and dropouts
python3.12 -m v2dlidar.visualize_lidar_scan --root ./dataset --scenario_id 0 --scan_id 0

# Raw wall segments (geometry used by LiDAR and planner)
# Left: existing room/door layout (apartment view)
# Right: raw wall segments (optionally overlaid on rooms/doors)
python3.12 -m v2dlidar.visualize_segments --root ./dataset --scenario_id 0 --side_by_side

# Segments only (no room overlay)
python3.12 -m v2dlidar.visualize_segments --root ./dataset --scenario_id 0 --segments_only

```

**Options:**
- `--root`: dataset root containing `scenario_XXXX/` folders (default: `./dataset`).
- `--scenario_id`: which scenario to load (e.g. `0` for `scenario_0000`).
- `--scan_id`: optional; overlays that scan's ideal path and start/goal from `scan_paths/scan_XXXX.json`.
- `--no_pose`: omit the LiDAR pose arrow overlay.

For `visualize_lidar_scan`:
- It always reads the per-scenario LiDAR spec from `scenario_meta.json` (`lidar` block), so `fov_deg`, `num_rays`, `max_range`, `range_noise_std`, and `dropout_prob` can vary between scenarios.
- The left subplot reuses the layout drawing from `visualize_scenario` in **world coordinates** and overlays:
  - The LiDAR pose for the selected scan.
  - All noisy rays (world-frame lines) and their ideal noise-free hit points.
  - The ideal A* path and start/goal (if available in `scan_paths/scan_XXXX.json`, unless `--no_path` is set).
- The right subplot (unless `--no_polar` is passed) shows a **robot-centric** polar plot of:
  - Beam angles **relative to the robotâ€™s heading** for that scan (0Â° = forward).
  - Noise-free ranges (baseline).
  - Noisy measured ranges for valid hits.
  - Dropouts plotted at `max_range`, making the effect of `dropout_prob` visually apparent.

For `visualize_segments`:
- It loads `ScenarioSpec` from `scenario_meta.json` and regenerates the scenario via `generate_scenario`, then plots the **raw wall segments** used by the LiDAR simulator and planner.
- With `--side_by_side`, it shows a figure with:
  - Left: the existing room/door layout view (using `visualize_scenario` logic).
  - Right: raw wall segments, optionally overlaid on rooms/doors (use `--no_rooms` to hide the overlay).
- With `--segments_only`, it shows only the segment lines for the chosen scenario, which is useful for directly checking which walls the LiDAR rays intersect.
